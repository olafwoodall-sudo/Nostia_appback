1. BLOCKING — Nothing ships without these

1A. Audit Git History for Committed Secrets

Why: If .env was ever committed, all credentials are compromised regardless of what the file contains today.

# Run from c:\nostia-app\
git log --all --full-history --oneline -- .env
git log --all --full-history --oneline -- "**/.env"
If any output appears: immediately rotate JWT_SECRET, STRIPE_SECRET_KEY, and STRIPE_WEBHOOK_SECRET in both Stripe Dashboard and DigitalOcean. Git history rewriting alone is not sufficient — treat keys as burned.

Why: The .env file must not be the live source of truth in production. Secrets belong in DO as encrypted vars.

1C. DigitalOcean — Persistent Database Volume

Why: DigitalOcean App Platform uses ephemeral storage. Every redeploy wipes the SQLite database. This is why user data disappears on every push.

Where: DigitalOcean Dashboard → Apps → nostia → Settings → Storage

  1. Click Add Volume
  2. Mount path: /app/database
  3. Size: 5–10 GB
  4. Save → Redeploy
Longer term: Consider migrating to DigitalOcean Managed PostgreSQL (~$15/mo) for backups, point-in-time recovery, and no ephemeral storage risk. Requires migrating the Node backend from better-sqlite3 to the pg package.

1D. Android Release Keystore

Why: nostia-release.jks does not exist. No signed APK/AAB can be built without it, and Play Store requires it.

CRITICAL: Back this file up permanently. Losing it locks you out of Play Store updates forever — there is no recovery. 
# Run once in c:\nostia-app\
keytool -genkey -v `
  -keystore nostia-release.jks `
  -alias nostia `
  -keyalg RSA -keysize 2048 `
  -validity 10000

When prompted: enter your name/org/location, choose a strong keystore password and key password.

After generating:

  1. Update nostia-android/local.properties: 
    KEYSTORE_FILE=../nostia-release.jks
KEYSTORE_PASSWORD=<your_password>
KEY_ALIAS=nostia
KEY_PASSWORD=<your_password>
  2. Back up nostia-release.jks to: password manager + encrypted cloud + physical USB
  3. Get the SHA-1 fingerprint (needed for step 1E): 
    keytool -list -v -keystore nostia-release.jks -alias nostia
# Copy the SHA1: line from the output

1E. Google Maps API Key

Why: Both nostia-expo/app.json (line 42) and nostia-android/local.properties contain placeholder values. The Friends Map screen will crash at runtime.

    • nostia-expo/app.json line 42: replace PLACEHOLDER_GOOGLE_MAPS_API_KEY
    • nostia-android/local.properties: replace your_google_maps_key

2. HIGH PRIORITY — Fix before store submission

2A. Stripe Webhook Verification

Why: The backend handles disputed charges and failed payments via webhooks. If the events aren't enabled, these handlers never fire.

Where: dashboard.stripe.com/webhooks

  1. Find or create endpoint: https://king-prawn-app-44tki.ondigitalocean.app/api/stripe/webhook
  2. Verify these events are enabled:
    • payment_intent.succeeded
    • payment_intent.payment_failed
    • charge.dispute.created
    • transfer.created
  3. Confirm the Signing Secret matches the value in DigitalOcean env vars

2B. SSL Certificate Pinning — Android

Why: Without pinning, a MITM on a user's network can intercept JWT tokens and payment intents.

Step 1 — Get the certificate pin hash (run in Git Bash or WSL):

openssl s_client -connect king-prawn-app-44tki.ondigitalocean.app:443 2>/dev/null \
  | openssl x509 -pubkey -noout \
  | openssl pkey -pubin -outform der \
  | openssl dgst -sha256 -binary \
  | base64

Step 2 — Add to OkHttp client in nostia-android/:

val pinner = CertificatePinner.Builder()
  .add("king-prawn-app-44tki.ondigitalocean.app", "sha256/<hash_from_above>")
  .build()
OkHttpClient.Builder().certificatePinner(pinner).build()
If the DigitalOcean domain or SSL certificate ever changes, you must ship a new app version with an updated pin. Document the hash somewhere safe.

2C. SSL Certificate Pinning — iOS

Why: nostia-ios/NostiaApp/Network/APIClient.swift uses URLSession.shared with no pinning.

Step 1 — Use the same hash from step 2B.

Step 2 — In Xcode: File → Add Package Dependencies → https://github.com/datatheorem/TrustKit

Step 3 — Configure in NostiaApp.swift before app launch:

import TrustKit
TrustKit.initSharedInstance(withConfiguration: [
  kTSKPinnedDomains: [
    "king-prawn-app-44tki.ondigitalocean.app": [
      kTSKPublicKeyHashes: ["<hash_from_step_2B>"],
      kTSKEnforcePinning: true
    ]
  ]
])

2D. Jailbreak Detection — iOS

Where: nostia-ios/NostiaApp/NostiaApp.swift

  1. In Xcode: File → Add Package Dependencies → https://github.com/securing/IOSSecuritySuite
  2. Add at app launch: 
    import IOSSecuritySuite
if IOSSecuritySuite.amIJailbroken() {
  // Warn user — do NOT hard-block (accessibility concerns)
  // Disable Vault / payment features only
}

2E. Root Detection — Android

Where: Application class in nostia-android/

  1. Add to app/build.gradle: 
    implementation 'com.scottyab:rootbeer-lib:0.1.0'
  2. Add in Application.onCreate(): 
    val rootBeer = RootBeer(this)
if (rootBeer.isDeviceRooted) {
    // Log event, warn user, disable Vault/payment features
}

2F. JWT Refresh Token Architecture

Why: Current JWTs last 7 days. Financial apps should use 15–60 min access tokens + long-lived refresh tokens.

Required changes:

  1. Backend (middleware/auth.js, server.js): new POST /api/auth/refresh endpoint; new refresh_tokens table; access token TTL reduced to 15–60 min; refresh token TTL 30 days with rotation
  2. Expo (api.ts): intercept 401 → call refresh → retry original → logout on refresh failure
  3. iOS (APIClient.swift): same pattern
  4. Android: same pattern in Retrofit/OkHttp interceptor
This is an architectural change. Plan a maintenance window and test all three clients before deploying.

2G. Expo Project ID for Push Notifications

  3. Update nostia-expo/src/services/notifications.ts — replace 'your-project-id'

4. MEDIUM PRIORITY

4A. EAS Code Signing (Expo OTA Updates)

Only needed if you plan to push JavaScript updates without going through the app store review process.

cd c:\nostia-app\nostia-expo
npm install -g eas-cli
eas login
eas update:configure
eas credentials

This generates a code signing certificate. Without it, a MITM can push arbitrary JS to all your users via the OTA channel.

4B. Inactivity Timeout

Why: The app handles real Stripe payments. If a user walks away from their unlocked phone, anyone can approve a vault expense.

On all three platforms: after 15 minutes in background, require biometric or PIN re-authentication before accessing Vault or initiating any payment. Does not need to be a full logout.

4C. Privacy Policy

Why: Required by both app stores, and legally required by GDPR/CCPA for apps collecting location + analytics data.

Must cover:

  • Data collected: name, email, location, photos, financial transactions
  • How it is used: no sale to third parties, analytics only within app
  • How to request deletion: already implemented in backend at DELETE /api/user
  • Contact email
  • Data retention: 90 days (matches current .env setting)

Publish at a stable public URL (GitHub Pages or custom domain). Link from both app store listings and in-app Settings screens.

4D. Custom Domain (Optional)

The app currently runs on king-prawn-app-44tki.ondigitalocean.app. For a branded API URL:

  1. Register a domain (e.g. api.nostia.app)
  2. DigitalOcean App Platform → Settings → Domains → Add Domain
  3. Add CNAME at your registrar pointing to the DO app URL
  4. DO auto-provisions Let's Encrypt SSL
  5. Update ALLOWED_ORIGINS env var + API base URLs in all clients
  6. Regenerate SSL pin hash (steps 2B/2C) after domain change
